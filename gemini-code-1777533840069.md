# Task: Office Staff Portal, Dynamic Fee Management, & Notification Sync

## Project Context
St. Francis De Sales Secondary School Student Management Application.
Tech Stack: HTML, CSS (Bento Grid, Glassmorphic UI), Vanilla JavaScript, Firebase v9+ (Auth & Firestore).

## Established Coding Rules & Constraints
1. **Modular Architecture:** Do not alter or interfere with the existing application routing logic.
2. **UI/UX Preservation:** Maintain the existing Bento Grid layout and glassmorphic CSS styling.

## Required Implementations

### 1. Role-Based Visibility (Office Staff Portal)
* Implement routing/visibility logic so that the Office Staff Portal is exclusively accessible to users with the "Office Staff" role in Firestore.
* Staff must have full read access to all student details to facilitate fee processing.

### 2. Class-Based Fee Structure & Dynamic Fees Card
* **Database Logic:** Establish a Firestore structure where a "Total Annual Fee" is defined per Class (e.g., Class 9 = ₹20,000, Class 10 = ₹25,000).
* **UI Component:** Create a dynamic "Fees Card" within the Staff Portal (maintaining the existing Bento Grid/Glassmorphic design).
* **Student Tracking:** When a staff member selects a student, the system should pull the total fee for their class and display the current remaining balance.

### 3. Payment Processing & Successive Deduction
* Add an input form within the Fees Card for staff to log new payments made by parents.
* **Math Logic:** Each logged payment must subtract from the student's current total fee balance in the UI temporarily until approved.
* **Database Write:** Save each transaction as an individual document in a `transactions` subcollection under the student, including timestamp, amount paid, and the ID of the staff member who logged it. Set the initial `status: "pending"`.

### 4. Printable Receipts
* Upon successfully logging a payment, generate a printable receipt component.
* Provide a vanilla JavaScript function using `window.print()` targeting just the receipt element. It should include the school name, student details, amount paid, date, and remaining balance.

### 5. Admin Approval Workflow
* **Admin View:** Create a read-only confirmation feed for the "Admin" role. 
* The Admin must be able to review pending transactions and click "Approve", which updates the transaction status to `status: "approved"` in Firestore.

### 6. Student Notification & Balance Sync (Post-Approval)
* **Trigger:** Once the Admin successfully clicks "Approve", trigger a write to the individual student's notification subcollection (integrating with the existing in-app notification center).
* **Notification Content:** Generate a personalized alert structured as: "Thank you, your payment of ₹[Amount Paid] is completed. This is for your acknowledgement. Remaining Balance: ₹[New Balance]."
* **Ledger Update:** Ensure the student's official master remaining balance is permanently updated in Firestore only *after* this Admin approval occurs.

## Deliverables Expected
1. **Firestore Schema Outline:** Briefly explain the database structure for Classes, Students, Transactions, and Notifications.
2. **Vanilla JS:** The Firebase v9 functions to fetch balances, process payments, handle admin approvals, and trigger the student notification.
3. **HTML/CSS:** The markup for the dynamic Fees Card and Printable Receipt.
